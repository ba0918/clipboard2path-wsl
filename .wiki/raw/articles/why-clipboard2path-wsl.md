# なぜ clipboard2path-wsl が必要なのか

## 問題の背景

WSL2（Windows Subsystem for Linux 2）は Windows 上で Linux 環境を実行する強力な仕組みだが、**クリップボード経由の画像貼り付け**に根本的な課題がある。

Windows ホスト側でスクリーンショット（Win+Shift+S など）やブラウザからの画像コピーを行っても、WSL2 上のアプリケーションに画像として直接貼り付けできないケースが多い。一部の環境では Alt+V での貼り付けが動作するが、環境依存が大きく安定しない。

## 技術的な原因

WSL2 のクリップボード連携は WSLg（Windows Subsystem for Linux GUI）が仲介する。WSLg は Wayland プロトコルを経由して Windows 側のクリップボードを Linux 側に転送するが、以下の制限がある：

1. **画像フォーマットの制限**: `wl-paste --type image/png` が動作せず、`image/bmp` のみ利用可能な環境が存在する
2. **アプリケーション間の互換性**: 画像データをそのまま貼り付けられるアプリと、ファイルパスを期待するアプリが混在する
3. **変換の不在**: Windows 側の BMP クリップボードデータと Linux 側が期待する形式の間に自動変換レイヤーがない

## clipboard2path-wsl の解決策

この問題を「画像データ → ファイル保存 → パスをクリップボードにセット」という変換パイプラインで解決する：

1. `wl-paste --type image/bmp` で Windows クリップボードから BMP 画像を取得
2. BMP → PNG に変換（ファイルサイズ削減＋汎用性）
3. `/tmp/clipboard-{timestamp}.png` に保存
4. `wl-copy` で保存パスをクリップボードにテキストとして書き戻す

これにより、ファイルパスを期待するあらゆるアプリケーションで画像を利用可能になる。

## 設計判断: なぜデーモンか

ワンショット実行（手動でコマンド叩く）でも問題は解決できるが、毎回手動実行は現実的ではない。常駐デーモンとしてクリップボードをポーリング監視し、画像を検知するたびに自動変換することで、ユーザーは意識せずにクリップボード画像をファイルパスとして利用できる。

## 設計判断: なぜ Rust か

- **バイナリサイズ**: ~600KB のシングルバイナリ。Go や Python に比べて桁違いに小さい
- **メモリ消費**: GC なし、常駐でも数 MB 程度
- **起動速度**: ミリ秒単位で起動、systemd サービスとして最適
- **安全性**: メモリ安全保証により、デーモンとしての長時間稼働が安心
