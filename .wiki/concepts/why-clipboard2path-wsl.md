# なぜ clipboard2path-wsl が必要なのか

## 概要

WSL2 環境ではクリップボード経由の画像貼り付けに制限があり、Windows 側でコピーした画像を Linux アプリケーションに直接貼り付けできないケースがある。clipboard2path-wsl はこの問題を「画像 → ファイル保存 → パスをクリップボードにセット」という変換パイプラインで解決する軽量デーモンである。

## 問題

WSL2 のクリップボード連携は [[WSL2 クリップボード連携の技術詳細|WSLg]] が仲介するが、以下の制限が存在する：

- **画像フォーマット制限** — `wl-paste --type image/png` が動作せず BMP のみ利用可能な環境がある
- **アプリ間互換性** — 画像データを直接受け付けるアプリと、ファイルパスを期待するアプリが混在
- **変換レイヤーの不在** — Windows 側 BMP と Linux 側が期待する形式の間に自動変換がない

Alt+V による貼り付けは一部環境で動作するが、環境依存が大きく安定しない。

## 解決策

clipboard2path-wsl は以下の4段階パイプラインで問題を解決する：

1. `wl-paste --type image/bmp` で Windows クリップボードから BMP 画像を取得
2. BMP → PNG に変換（ファイルサイズ削減＋汎用性向上）
3. `/tmp/clipboard-{timestamp}.png` に保存
4. `wl-copy` で保存パスをクリップボードにテキストとして書き戻す

これにより、ファイルパスを期待するあらゆるアプリケーションで画像を利用可能になる。

## 設計判断

### なぜデーモンか

手動実行でも問題は解決できるが、毎回コマンドを叩くのは非現実的。常駐デーモンとしてクリップボードをポーリング監視し、画像検知時に自動変換することで、ユーザーの操作なしにクリップボード画像をファイルパスとして利用できる。

### なぜ Rust か

| 観点 | Rust の利点 |
|------|------------|
| バイナリサイズ | ~600KB のシングルバイナリ |
| メモリ消費 | GC なし、常駐でも数 MB |
| 起動速度 | ミリ秒単位、systemd 向き |
| 安全性 | メモリ安全保証で長時間稼働が安心 |
